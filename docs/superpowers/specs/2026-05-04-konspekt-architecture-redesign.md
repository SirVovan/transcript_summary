# Спек: Пересмотр архитектуры скилла /konspekt

**Дата:** 2026-05-04  
**Статус:** На проверке у пользователя

---

## Проблема

Текущая архитектура имеет два фундаментальных изъяна:

1. **Транскрипт читается дважды.** ШАГ 1 читает транскрипт для построения карты. ШАГ 2 читает транскрипт снова (куски) при написании конспекта. Это удваивает стоимость и время.

2. **Вычисление и представление смешаны.** Картограф в одном шаге строит структуру И думает про XLSX-шаблоны. Конспектолог пишет текст И думает про DOCX-формат. Это раздувает инструкции и замедляет каждый шаг.

Реальный типичный транскрипт — 3–4 часа видео (~36–48K токенов). Пользователь вынужден вручную резать его на части перед запуском скилла.

---

## Цели

- Транскрипт читается ровно один раз
- Ручная нарезка транскрипта уходит из рук пользователя
- Контекст каждого LLM-вызова остаётся маленьким и фиксированным
- Качество не падает: логические переходы между сегментами сохраняются, голос спикера единый

---

## Новая архитектура: два слоя

```
СЛОЙ 1: Вычисление (производит мастер-MD)
СЛОЙ 2: Представление (рендерит из мастер-MD в форматы) ← отложен
```

Слой 2 проектируется отдельно. Текущий спек покрывает только Слой 1.

---

## Слой 1: Вычисление

### Обзор пайплайна

```
Транскрипт
    ↓
ШАГ 0: Предобработка (если транскрипт > 25K токенов)
    ↓
ШАГ 1а: Сегментация + самопроверка
    ↓
ШАГ 1б: Параллельные subagents (по одному на сегмент)
    ↓
ШАГ 1в: Бригадир (гармонизация + верификация)
    ↓
Мастер-MD
```

---

### ШАГ 0: Предобработка больших транскриптов

**Триггер:** транскрипт > ~25K токенов (примерно 2+ часа видео).

**Алгоритм (гибридный скрипт + LLM):**

1. Скрипт идёт по транскрипту и накапливает токены.
2. При достижении ~15–18K токенов вырезает окно ±10 минут (~2–3K токенов) вокруг текущей позиции.
3. Лёгкий LLM-вызов (минимальный промпт): *«В этом фрагменте найди точку, где заканчивается смысловой блок. Верни тайминг или первые слова следующего блока. Если не уверен — скажи об этом явно.»*
4. Скрипт режет по найденной границе с перекрытием ~500 токенов.

**Самопроверка:**
- Если LLM уверен → нарезка происходит молча.
- Если LLM выражает сомнение → пауза, пользователь получает уведомление (через расширение), объяснение и запрос подтверждения.

**Результат:** 2–4 части по ~15–18K токенов каждая. Каждая часть проходит через ШАГ 1а–1в независимо. В конце части объединяются в один мастер-MD.

---

### ШАГ 1а: Сегментация

**Вход:** один фрагмент транскрипта (~15–25K токенов).

**Процесс:**
1. Черновой прогон: Claude читает транскрипт и выделяет смысловые сегменты с метаданными (тайминг, тема, тип блока, спикер, краткая суть в буллетах).
2. Самопроверка по критериям:
   - Каждый сегмент — одна связная мысль (не обрывается, не смешивается с другой)?
   - Между соседними сегментами читается логический переход?
   - Нет аномалий по длине (сегмент < 2 мин или > 30 мин без явной причины)?
   - Последовательность сегментов отражает ход мысли автора?
3. Если все критерии выполнены → показать список сегментов пользователю (информационно), продолжить без ожидания ответа.
4. Если найдена проблема → объяснить что именно + запросить решение у пользователя.

**Профиль:** определяется в начале сессии. Пользователь объявляет тип контента (лекция / custdev / встреча / конференция). Если не объявил — Claude определяет автоматически по началу транскрипта.

**Результат:** список сегментов с метаданными (~1–2K токенов) — навигационная карта для следующих шагов.

---

### ШАГ 1б: Параллельные subagents

Для каждого сегмента запускается отдельный subagent. Все запускаются параллельно.

**Каждый subagent получает:**
1. Навигационную карту (список всех сегментов с метаданными, ~1–2K токенов)
2. ToV-спецификацию (см. ниже)
3. Свой срез транскрипта (только один сегмент)

**ToV-спецификация (гибридная):**
- Базовая часть: фиксированная в скилле (первое лицо, живая речь, без академизма, конкретные примеры перед тезисами).
- Динамическая часть: извлекается автоматически из первых ~2–3K токенов транскрипта перед запуском subagents. Claude выделяет: характерные обороты спикера, темп и уровень формальности, фирменные фразы и обращения к аудитории (~150–200 токенов).

**Каждый subagent производит** два блока для своего сегмента:

```markdown
### Карта
[буллеты: ключевые тезисы, аргументы, примеры сегмента]

### Текст
[связный текст от первого лица спикера]
```

**Subagent НЕ видит:** результаты соседних subagents, полный транскрипт, инструкции по форматам вывода.

---

### ШАГ 1в: Бригадир

Один Claude читает все результаты subagents + навигационную карту и выполняет три задачи.

**Задача 1 — Верификация качества каждого сегмента:**
- По навигационной карте проверить: все ключевые тезисы сегмента отражены в тексте?
- Нет ли явных пропусков, не соответствующих карте?

**Задача 2 — Гармонизация:**
- Выровнять стиль и голос по всем сегментам.
- Написать логические переходы между сегментами (1–2 предложения на стыке).

**Задача 3 — Исправление проблем:**
- Проблема стиля/связности → бригадир исправляет сам.
- Пропущен важный тезис из карты → отправить subagent на переделку с конкретным feedback: *«В сегменте X по карте должен быть тезис Y — он пропущен, переделай этот блок.»*
- Если второй прогон subagent тоже неудовлетворителен → поставить метку `⚠️ требует проверки` и продолжить.

**Результат:** финальный мастер-MD.

---

## Мастер-MD: формат

Единственный артефакт Слоя 1. Используется как источником для всех рендеров Слоя 2 и как контекст для будущих сессий (вместо перечитывания транскрипта).

```markdown
# [Название выступления]

**Спикер:** [имя]  
**Дата:** [дата]  
**Длительность:** [время]  
**Профиль:** [лекция / custdev / встреча / конференция]  
**Сегментов:** [N]

---

## Сегмент 1 | 00:00:00–00:14:30 | [Тема]

**Тип блока:** [метка]  
**Спикер:** [имя, если несколько спикеров]

### Карта
- [Ключевой тезис]
- [Аргумент / пример]
- ...

### Текст
[Связный текст от первого лица спикера. Несколько абзацев.]

---

## Сегмент 2 | ...
```

---

## Точки остановки

| Точка | Триггер | Действие |
|---|---|---|
| ШАГ 0 | LLM не уверен в точке реза | Пауза + уведомление + запрос |
| ШАГ 1а | Найдена проблема сегментации | Объяснение + запрос |
| ШАГ 1а | Всё ок | Показать карту, продолжить автоматически |
| После ШАГ 1в | Опционально (режим с проверкой) | Показать мастер-MD, ждать «ок» |

Звуковое уведомление при паузах — через расширение пользователя (не требует изменений в скилле).

---

## Что убирается из архитектуры

- **XLSX-формат** — убирается полностью.
- **DOCX-формат** — убирается полностью.
- **Карта-MD как отдельный артефакт Слоя 1** — убирается. Если нужна карта без текста — это рендер Слоя 2.
- **Двойное чтение транскрипта** — устраняется.
- **Ручная нарезка больших транскриптов** — автоматизируется в ШАГ 0.

---

## Отложено (не входит в текущую реализацию)

- **Слой 2: Рендеры** — HTML-виджет, карта-MD, возможные форматы (MindMap и др.). Проектируется отдельно после реализации Слоя 1.
- **Детальный дизайн профилей** — базовое поведение (объявление пользователем / автоопределение) реализуется сейчас. Специфика каждого профиля — отдельный спек.
- **Многочастевая сборка** — детали объединения нескольких частей в единый мастер-MD при обработке очень длинных транскриптов.
