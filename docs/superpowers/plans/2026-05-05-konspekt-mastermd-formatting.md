# План: Улучшение форматирования мастер-MD

**Дата:** 2026-05-05  
**Статус:** готов к реализации  
**Контекст:** После валидации новой chunk-based архитектуры (задачи 1–7 выполнены) выявлены улучшения формата мастер-MD.

---

## Проблемы, которые решаем

1. **Нет форматирования в Тексте.** Текст идёт сплошным нарративом без выделений — глазу не за что зацепиться. Ключевые термины, главные мысли, имена инструментов не подсвечены.

2. **Нет специальных блоков.** Лайфхаки, практические советы, демонстрации ничем не выделены — растворяются в общем тексте.

3. **Карта «богаче» Текста.** Буллеты Карты — ёмкие и структурированные, а в Тексте соответствующий материал не всегда раскрыт полноценно. Причина: субагент не получает явного требования раскрыть каждый буллет. Решение — жёсткое требование на входе, не только контроль на выходе.

4. **Нет «Ключевой мысли».** Читатель видит заголовок сегмента, но не знает сразу главного тезиса. Нужна 1–2 строки до Карты.

## Задача 1: `subagent_phase2.md`

Добавить в раздел «Что произвести» четыре изменения:

### 1а. Блок «Ключевая мысль» (перед Картой)

Субагент пишет 1–2 предложения — суть сегмента одной мыслью. Ставится между метаданными сегмента и блоком `### Карта`.

```markdown
**Ключевая мысль:** [1–2 предложения — самое главное, что можно унести из этого сегмента]
```

### 1б. Форматирование в Карте

В каждом буллете Карты жирным выделять ключевой термин или главный субъект пункта:

```markdown
- **Режим ByPass Permissions:** по умолчанию отключён; включается через settings.json или промт
- **Рекомендованные расширения VS Code:** Claude Status Bar, Office Viewer, Claude Notifier
```

### 1в. Форматирование в Тексте

Правила для блока `### Текст`:
- **Ключевые термины** (инструменты, названия, понятия) выделять жирным при первом упоминании в сегменте
- **Главную мысль абзаца** выделять жирным (обычно первое или последнее предложение абзаца)
- Нумерованные списки использовать для пошаговых инструкций, если они есть в транскрипте

### 1г. Специальные блоки (внутри Текста)

Три типа блоков, используются по контексту:

```markdown
> **Лайфхак:** [практический совет, который можно применить сразу]

> **Демонстрация:** [описание того, что спикер показывает на экране или делает вживую]

> **Важно:** [предостережение, ограничение, критически важная деталь]
```

Субагент сам определяет, какой тип подходит — только если в транскрипте явно есть такой материал. Не добавлять блоки искусственно.

### 1д. Жёсткое требование Карта → Текст

Добавить в правила блока `### Текст`:

> Каждый буллет из блока Карта должен быть содержательно раскрыт в Тексте — развёрнут как абзац или несколько предложений, а не просто упомянут вскользь. Если буллет Карты не находит развития в Тексте — это ошибка субагента.

## Задача 2: `tov_spec.md`

Добавить новый раздел **«Форматирование»** перед разделом «Динамическая часть».

Содержание раздела:

```markdown
## Форматирование

### Жирный текст

- **Ключевые термины** (инструменты, названия, концепции) — жирным при первом упоминании в сегменте
- **Главная мысль абзаца** — жирным (первое или последнее предложение)
- В Карте: ключевой термин или субъект каждого буллета — жирным

Не злоупотреблять: жирный работает только если используется редко. 1–2 выделения на абзац — максимум.

### Специальные блоки

Три типа блоков для особого контента:

> **Лайфхак:** практический приём, применимый сразу

> **Демонстрация:** описание того, что спикер показывает вживую

> **Важно:** предостережение, ограничение, критическая деталь

Блоки использовать только при наличии явного материала в транскрипте.

### Списки

Нумерованные списки — для пошаговых инструкций (если есть в транскрипте).
Маркированные списки — для перечислений без строгого порядка.
Не объединять пункты, не сокращать ради экономии строк.
```

## Задача 3: `brigadir.md`

Два точечных изменения:

### 3а. Шаблон сборки мастер-MD

В секции «Сборка мастер-MD» добавить строку `**Ключевая мысль:**` в шаблон сегмента:

```markdown
## Сегмент 1 | HH:MM:SS–HH:MM:SS | [Тема]

**Тип:** [риторическая роль]
**Спикер:** [имя или «основной»]

**Ключевая мысль:** [1–2 предложения]

### Карта

- [буллет]

### Текст

[текст]
```

### 3б. Верификация (страховочная проверка Карта ↔ Текст)

В секции «Верификация текста» добавить дополнительную проверку после существующей:

> После проверки тезисов чистовой карты: по каждому буллету блока `### Карта` убедиться, что в блоке `### Текст` есть содержательное раскрытие (абзац или несколько предложений), а не просто упоминание. Если буллет не раскрыт — запрос на переделку по той же схеме.
