## Контекст: текущая архитектура

Пайплайн строго последовательный, три шага:

```
Транскрипт (~16–24K токенов)
    ↓
ШАГ 1: Карта смыслов  (~60 KB инструкций)
    Читает: kartograf.md (26.5 KB) + профиль (0.9–4.6 KB) + xlsx_template.py (14.7 KB) + karta_example.html (18.6 KB)
    Выход: .md + .xlsx + .html  (~3–4K токенов карта)
    ↓
ШАГ 2: Конспект  (~25 KB инструкций)
    Читает: konspektolog.md (14.1 KB) + konstitutsiya.md (10.5 KB)
    Источник: карта смыслов .md как навигатор + куски транскрипта
    Выход: DOCX или внутренний текст
    ↓
ШАГ 3: Виджет  (~10 KB инструкций, после оптимизации май 2026)
    Читает: widget.md (10.2 KB)
    Выход: JSON → widget_generator.py → HTML
```

Всё выполняет один Claude в одном контексте, последовательно.

---

## Цели оптимизации

1. **Сократить время** — получать результат быстрее
2. **Сократить токены** — снизить стоимость API
3. **Не потерять качество** — ничего важного из лекции не теряется

Все три цели одинаково важны.

## Ключевые наблюдения (собраны перед брейнштормингом)

**Карта смыслов — центральный артефакт:**
- Содержит структуру, логику, тайминги всего выступления
- ~3–4K токенов против ~20K транскрипта
- Используется как навигатор на ШАГ 2 — именно поэтому конспект не пишется прямо по транскрипту

**Почему ШАГ 2 — главный кандидат для subagents:**
- Карта смыслов обеспечивает координацию (каждый subagent видит полную структуру)
- Блоки конспекта в основном независимы
- Связь с соседними блоками: достаточно резюме предыдущего (передаётся в промпт)
- Потенциальная схема: `N сегментов → N параллельных subagents → склейка`

**Открытые вопросы для брейнсторминга:**

1. **ШАГ 1 — можно ли его ускорить?**
   - Карта строится последовательно (сегментация → самопроверка → финальная таблица)
   - Есть ли смысл параллелить отдельные части?

2. **ШАГ 2 — subagents: как именно?**
   - Каждый сегмент = отдельный subagent?
   - Или группы сегментов (по теме/типу)?
   - Как передавать связь с предыдущим?
   - Как склеивать результаты?
   - Кто управляет: основной Claude или orchestrator?

3. **ШАГ 3 — есть ли потенциал?**
   - JSON пишется линейно, виджет собирается скриптом
   - Можно ли параллелить генерацию JSON по сегментам?

4. **Общая архитектура:**
   - Нужен ли orchestrator-агент поверх трёх шагов?
   - Как хранить промежуточные результаты между subagents?
   - Что если subagent «провалится» (плохое качество одного блока)?

5. **Трейдоффы subagents:**
   - Каждый subagent читает инструкции заново → токены множатся на N
   - Параллельность ускоряет время, но НЕ экономит токены (часто наоборот)
   - При каком количестве сегментов subagents становятся выгодны по времени?

## Направления для обсуждения

**А. Гибридная архитектура:**
Основной Claude делает ШАГ 1 (карта неделима). На ШАГ 2 — запускает N subagents параллельно (по сегменту каждый). Склейка + ШАГ 3 — снова основной.

**Б. Полный orchestrator:**
Отдельный управляющий агент координирует всё. Три специализированных агента: картограф, конспектолог, виджетолог. Каждый получает только своё.

**В. Оставить последовательно, но оптимизировать иначе:**
Вместо subagents — prompt caching (неизменяемые reference-файлы кэшируются, ~10% стоимости при повторных запросах). Плюс дальнейшее сокращение reference-файлов (направление А аудита).

## Что НЕ обсуждалось (вопросы для новой сессии)

- Приемлемо ли увеличение токенов ради параллельности (время vs деньги)?
- Как обрабатывать ошибки отдельных subagents?
